## The problem

Claude Code's first turn is reliably:

```
Glob â†’ Grep â†’ Read â†’ Read â†’ Read â†’ "found it!"
```

before it actually does anything useful. On a 6-prompt fixture I instrumented with `claude --print --stream-json`, the control arm averaged **51,000 tokens per prompt**, of which roughly **35k was first-turn exploration**.

The model has no map of the codebase going into turn 1. So it grep-walks one. Every Claude Code user pays this cost on every fresh session, regardless of model or pricing tier.

I kept hitting the 5-hour rate window mid-refactor, so I built a map.

## The fix

Three pieces, all stdlib Python:

### 1. `build_repo_graph.py` â€” install-time, ~1s on a 5k-file repo

Walks the source tree, extracts:
- **Symbols**: function/class/struct/interface/type definitions via per-language regex (Py, JS/TS, Rust, Go).
- **Import edges**: who imports whom.
- **Hot files**: top files by `git log --since=90d` change frequency.
- **Path DF**: precomputed document frequency for IDF weighting.

Writes `.context-os/repo-graph.json`. ~67KB on this repo (36 files, 444 symbols).

### 2. `auto_context.py` â€” UserPromptSubmit hook, ~50ms

Reads the user's prompt off stdin. Extracts identifier and path tokens, expands NL â†’ code (`authentication â†’ auth`, `matcher â†’ match`), generates pluralâ†”singular variants. Scores every (file, symbol) candidate against the graph using:

- `symbol_exact` â€” token equals symbol name (IDF-weighted)
- `symbol_ci` â€” case-insensitive match
- `path_exact` â€” token equals basename
- `path_substr` â€” token is substring of path (df-discriminative bonus: `disc = 1 - df/N`)
- `import_edge` â€” file imports a symbol-bearing file
- `hot_file` â€” boost for git-hot files
- `test_penalty` / `hub_penalty` â€” penalize over-broad files
- `multi_token_coverage` â€” bonus when multiple distinct prompt tokens hit one file

Aggregates per-file (sum candidate scores, pick best-scoring line as representative), takes top-N, prepends a compact `<context-os:autocontext>` block to the prompt before Claude sees it.

```
<context-os:autocontext>
crates/ignore/src/gitignore.rs:42  Â· Gitignore     (struct)
crates/ignore/src/gitignore.rs:118 Â· matched       (fn) Â· imports: glob, walk
crates/ignore/src/walk.rs:201      Â· WalkBuilder   (struct)
</context-os:autocontext>
```

### 3. `prewarm.py` â€” SessionStart hook

Detects a stale graph (>7 days old or >20 source files newer) and rebuilds in the background via detached `subprocess.Popen`. User's next prompt uses the fresh index.

That's it. Three Python files, all stdlib, ~600 lines combined.

## Why no embeddings?

Three reasons:

1. **Cold start.** Embedding 50k files takes minutes and needs a model loaded. The Python hook ships with `python3` only.
2. **Cost.** Every prompt becomes a model call before Claude sees it.
3. **Quality.** On top-1 precision (the only metric that matters because Claude only acts on the top result), well-tuned BM25 + path heuristics + a few semantic-ish signals already get MRR 0.984 on synthetic. Embeddings help on broad recall, which isn't the bottleneck.

If/when this gets ported into `claude` itself, embeddings as an optional reranker on top of the lexical layer is the obvious v3.

## Why this should be inside `claude` itself

Discovery is the problem. Users who would benefit most (new to Claude Code) won't find a GitHub plugin. The savings only accrue to Anthropic's hosted users if the primitive is on by default.

Three integration paths, ordered by depth (full detail in the [pitch doc](https://github.com/sravan27/context-os/blob/main/docs/PITCH.md)):

| Path | Anthropic-side cost | User-side cost | Win fraction |
|---|---|---|---|
| **(A) Bundle the hook** | Zero (donate) | One opt-in command | Power users only |
| **(B) First-class primitive** (`claude context build`) | ~1 engineer-week port to Rust | Zero (auto) | Most users |
| **(C) Default-on with telemetry-driven rebuild** | Anthropic-scale measurement | Zero | All users |

I recommend (B). MIT-licensed; happy to PR or donate.

## Honest limitations

- **Not Anthropic-scale.** 36 live calls on 6 prompts is a proof of concept, not production telemetry.
- **Not universal.** On repos where prompts already name the exact class (psf/requests in cross-repo), bm25-symbols matches us. We win the aggregate, not every repo.
- **Not a replacement for embeddings on broad-recall tasks.** I target precision at the first candidate, so the first `Read` is correct.
- **Tested on Py/TS/Rust/Go.** Other languages fall back to path-only ranking.
